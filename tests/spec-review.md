# Functional Spec Review — Unclear / Ambiguous Specifications

| Page | Comment | Justification |
|---|---|---|
| **Sign In** | No input fields are specified in the functional spec. Section 3 states "No data is required on this page" and section 4.1 lists only "Application name" and "Sign in action" — no email, password, or credential fields. | The Page Layouts reference clearly shows an Email field and Password field on the sign-in card. The functional spec and the Page Layouts design reference contradict each other. |
| **Sign In** | The error message on failed sign-in is not defined. "On failure, an error message is shown" — but no message text, placement, or failure conditions (wrong password, unknown user, etc.) are described. | Without a defined error message, implementations will vary; this is a testable requirement that cannot be verified. |
| **Calendar Overview** | A "Request Leave" button visible to Employees is present in the Page Layouts reference but is entirely absent from the Calendar Overview functional spec elements table. | Page Layouts explicitly lists it as a page element ("Primary variant, top-right of the page header. Not visible to Admins."). This is a whole interactive element missing from the spec. |
| **Calendar Overview** | "Truncated with overflow indicator if more than a set number fit" — the threshold ("a set number") is never defined, and the overflow indicator behavior ("+N more"? expandable? tooltip?) is not described. | Without a defined threshold and overflow UX, implementations will differ and cannot be tested deterministically. |
| **My Leave** | Employees can register leave starting today ("From: must be today or a future date") but can only edit or delete leave with a **future** start date. This means a registration for today cannot be edited or deleted once submitted. | This is likely unintentional — a user who registers today's leave by mistake immediately loses the ability to correct it. If intentional, the business reason should be stated. |
| **My Leave** | The "Duration (days)" column is shown in the table but no calculation rule is defined anywhere — neither here nor in the Leave Registration data model (which has no duration attribute). | Is duration inclusive of both start and end date? Calendar days or working days? The spec is silent. This is a derived field that needs a defined formula. |
| **My Leave / Leave Management** | The leave registration form (Register/Edit) is not specified as a modal, side panel, or separate page. Submit/Cancel buttons are described but the rendering context is never stated. | The Confirmation Dialog explicitly states it is a "modal overlay". The equivalent absence of this specification for the leave form is a gap that affects both design and implementation. |
| **My Leave** | The button is labelled "Register leave button" in the functional spec, but the Page Layouts reference calls it "Request Leave". | The label determines the actual text shown on-screen and in test specs; these must match. |
| **Leave Management** | The functional spec places no date restriction on admin edit/delete actions. The Page Layouts reference says the Actions column (Edit + Delete) is "only shown for future-dated registrations" on Leave Management — the same constraint as for employees. These contradict each other. | The spec should explicitly state whether admins can edit/delete historical leave or only future leave. |
| **Leave Management** | The admin Add/Edit form has no stated constraint on the From date (no "today or future" rule). It is never explicitly stated that admins *can* register past-dated leave. | The omission leaves implementers to guess. If admins are intentionally allowed to register past leave (e.g. for sick leave recorded retrospectively), the spec should say so explicitly. |
| **Leave Management** | The "Reset filters" ghost button visible in the FilterBar is documented in the Page Layouts and Component Library references but is entirely absent from the Leave Management functional spec elements table. | The Page Layouts doc itself flags this in a Dutch-language note: "De functional spec noemt geen Reset-knop in de FilterBar." |
| **Navigation Bar** | The Design System Application Pages table shows `/my-leave` as available to "Employee" only, and the Component Library Nav Bar section lists Admin navigation as "Calendar Overview, Leave Management" (no My Leave). Yet the My Leave page spec says "Available to: Employee, Admin". | If admins cannot reach My Leave via the navigation bar, and there is no other documented route to it, admins are effectively locked out despite the page spec saying they have access. |
| **Employee Leave Chip** | Section 4.1 element 2 reads "The employee's first name or initials" — it is ambiguous whether the chip displays the first name, initials, or one as a fallback for the other. | First name ("John") and initials ("JS") are materially different representations. The spec should specify one deterministically. |
| **Employee Leave Chip / Leave Type Badge** | Both component pages list leave type colours as "[To be defined by design]", but the Design System has already defined them: Vacation #10B981, Sick Leave #F59E0B, Public Holiday #6366F1, Other #94A3B8. | The functional spec is out of sync with the Design System. Developers reading only the component pages will find the colour values missing, while they exist on a different page. One of the two should be updated. |
| **Confirmation Dialog** | Section 3 states "The button labels are fixed" (plural), but section 4.1 element 5 states "The button label reflects the action (e.g. 'Delete')". These are contradictory. | If the confirm button label is always fixed, it cannot reflect the action; if it varies per context, it is not fixed. The spec must clarify whether both labels are fixed or only the Cancel label is fixed while the confirm label is configurable. |
| **Data Model — Employee** | The Employee data model has no authentication attributes (email, username, password). The description says "Each employee has an associated identity used to authenticate them when signing in" but this identity is never modelled. | The Sign In page design (Page Layouts) shows email and password fields. It is unknown how authentication credentials relate to the Employee entity — whether they are separate (e.g. an auth provider) or attributes of Employee. |
| **Data Model — Leave Registration** | The overlap rule states registrations "may not overlap in date range", but the boundary condition is undefined. If Registration A ends on June 15 and Registration B starts on June 15, is that an overlap? | This is a common edge case. The form validation and backend both need to implement the same rule — the spec must define whether the boundary dates are inclusive or exclusive. |
| **Data Model — Leave Type** | The "Registerable by" attribute description says "Values: Employee, Admin" (implying a single-value enum), but the initial data table uses "Employee, Admin" as a combined value for three of the four types. It is unclear whether this is a single-value or multi-value field. | "Public Holiday: Admin only" vs "Vacation: Employee, Admin" — the latter appears to mean both roles can register it, implying multi-value, but the data model does not confirm this. |
| **Data Model — Leave Type** | "Whether leave types can be managed by an administrator or are a fixed list has not yet been determined." This is an explicitly open requirement. | If leave types are admin-manageable, this implies additional CRUD screens, API endpoints, and data model constraints (e.g. colour uniqueness enforcement) that are entirely unspecified. |
| **Functional Requirements** | No redirect-back behavior is specified for unauthenticated deep-link access. The Base Layout says unauthenticated users are redirected to Sign In, but it is unspecified whether the application preserves the original URL and redirects the user there after successful sign-in. | This is standard expected behavior in authenticated apps; omitting it means the spec does not cover the full authentication flow (e.g. a bookmarked `/admin/leave` link). |
